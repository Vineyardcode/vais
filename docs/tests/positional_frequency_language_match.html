Positional Frequency Matching Across Languages

Instead of assuming Latin VALUES for abbreviation marks, this test asks:
"For each candidate language, what character BEST MATCHES the positional
frequency profile of each EVA glyph?"

Method:
  1. Compute VMS positional profiles: for each EVA glyph, measure
     P(initial), P(final), P(medial), overall frequency
  2. Compute the same profiles for each candidate language corpus
  3. Find the optimal frequency-rank mapping: match EVA glyphs to
     target language characters by their positional frequency ranks
  4. Apply mapping to f1r and see which language produces the most
     recognizable output

This is NOT a decipherment — it's a statistical compatibility test.
The mapping is constrained only by positional frequency, not by
any assumed sound values.

Run it in your browser

Python + numpy on WebAssembly, entirely on your machine — nothing is uploaded. First run downloads the runtime (~15 MB) and the repository data pack (~9 MB); both are cached afterwards. Expect it to be slower than a native run.

────────────────────────────────────────────────────────────────────────────────
ASSESSMENT
────────────────────────────────────────────────────────────────────────────────

This test matches VMS glyphs to language characters purely
by POSITIONAL FREQUENCY — the most common word-final VMS glyph
maps to the most common word-final character in each language,
etc. No assumed sound values. No abbreviation meanings.

If the VMS uses a simple positional substitution of a known
language, this should produce recognizable text for the correct
language, even without knowing the specific key.

CRITICAL CAVEAT: This assumes 1-to-1 glyph-to-character mapping.
If the encoding is verbose (1 glyph → multiple chars) or
compressed (multiple glyphs → 1 char), this method cannot work.
It also treats EVA digraphs (ch, sh) as single units, which
may or may not be correct.
