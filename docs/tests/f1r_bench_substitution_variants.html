Aggressive bench substitution variants on f1r

Try substituting ch/sh benches with JKP's ligature candidates
to see if any combination produces recognizable text.

Run it in your browser

Python + numpy on WebAssembly, entirely on your machine вЂ” nothing is uploaded. First run downloads the runtime (~15 MB) and the repository data pack (~9 MB); both are cached afterwards. Expect it to be slower than a native run.

The abbreviation substitution produces NO recognizable continuous
text in any variant. Specific problems:

1. BENCH OPACITY: ch and sh appear in ~40% of tokens but have
   no single definitive substitution. They could represent 5+
   different Latin ligatures each, and the 'correct' mapping may
   vary by position and context.

2. GALLOWS MYSTERY: k, t, p, f (especially benched variants cth,
   cph, cfh) are composite constructions. Our simple kв†’I mapping
   is almost certainly wrong for most contexts.

3. THE 'o' PROBLEM: EVA-o appears in 55 bigrams on f1r alone.
   Whether it's a vowel, null, separator, or something else
   fundamentally changes the decoded output.

4. ADDITIONAL CIPHER LAYER: If abbreviation is only ONE component
   of a multi-layer encoding (as cappelli_abbreviation_retest suggested at ~30%),
   then abbreviation substitution alone cannot produce readable
   text вЂ” there's still an alphabet substitution or verbose
   cipher layer underneath/on top.

5. WRONG UNIT SIZE: JKP argues some EVA characters are 2-4
   components. If EVA character boundaries are wrong, ALL our
   substitutions are operating on incorrect units.

CONCLUSION: We do NOT have enough information to translate the
manuscript. The abbreviation mappings explain the STRUCTURAL
properties (paradigm fill, suffix distribution) but do not
constitute a decipherment key. A successful decoding would need:
  - Correct character segmentation (not necessarily EVA)
  - Resolution of bench ambiguity (ch, sh в†’ which ligature?)
